Protocol K

이슈 퍼스트 워크플로 · 자동화 스킬 · 브랜치 명명 · PR 정책 · 리뷰/머지 룰 · 코드 품질.

Workflow

이슈 퍼스트 워크플로

이슈 없는 코드 없음. 모든 작업은 GitHub Issue 생성 → 자동 브랜치 생성 → 개발 → PR → 머지 흐름을 따릅니다.

① 이슈 생성 Claude / Codex 명령
→
② Sprint 배정 자동
→
③ 브랜치 생성 Actions 자동
→
④ 개발 · PR · 머지 본인 진행

한 줄 명령어로 ①~③ 모두 자동

Claude Code 또는 Codex CLI 에서:

"이슈 생성해줘 — XX 기능 추가 (Feature)"

이 한 줄이 다음을 모두 수행합니다.

  1. GitHub Issue 생성 — 이슈 템플릿(feature-task.yml) 적용. 제목·본문·브랜치 유형 자동 wrap.
  2. 현재 Sprint에 자동 배정 — assign-sprint.sh가 오늘 날짜 → Sprint 매핑.
  3. GitHub Actions가 본문 파싱 → 브랜치 자동 생성 — auto-branches-from-issue.yml이 feature/<slug>-issue 형태로.
  4. (선택) 로컬에 worktree로 즉시 체크아웃 — checkout-issue-worktree.sh로 격리된 디렉토리 준비.
Automation Skill

.claude / .codex 둘 다 동작

어떤 도구로 작업해도 동일 결과. SKILL.md만 도구별로 별도이고, 스크립트는 공유합니다.

도구SKILL 위치호출 방식
Claude Code.claude/skills/create-issue/SKILL.mdAskUserQuestion 등 Claude 도구
Codex CLI.codex/skills/create-issue/SKILL.mdrequest_user_input 등 Codex 도구

공유 스크립트

스크립트역할
scripts/wrap-issue-body.sh이슈 템플릿 헤더로 본문 wrapping (헤더 이모티콘·공백·줄바꿈 보존)
scripts/assign-sprint.sh현재 Sprint 자동 배정 (날짜 비교 + project item 조회 + mutation)
scripts/checkout-issue-worktree.sh연결된 브랜치를 worktree 또는 일반 체크아웃으로 준비
최초 셋업 필요: assign-sprint.sh는 GitHub Project ID·Sprint Field ID·Sprint 표가 채워져야 동작합니다. 가이드: docs/setup/sprint-setup.md
Branch Naming

브랜치 명명 규칙

이슈 본문의 ### 🌿 브랜치 유형 값을 기반으로 GitHub Actions가 자동 생성합니다.

Prefix용도예시
feature/<slug>-issue신규 기능 · 설정 · 구성feature/add-mcp-gateway-issue
fix/<slug>-issue일반 버그 수정fix/auth-key-encoding-issue
hotfix/<slug>-issue운영/배포 영향 긴급 수정hotfix/gateway-500-issue
chore/<slug>-issue문서 정리 · 리팩토링 · 기능 변경 없음chore/update-readme-issue

이름 충돌 시

같은 slug 가 이미 존재하면 자동으로 순번이 붙습니다 (-01-issue, -02-issue …). 최대 20개.

Base 브랜치

  • → 자동 생성 브랜치의 base = dev
  • → PR 머지 흐름: feature/* → dev → main
PR Policy

PR 리뷰·머지 컨벤션

팀 구성: 총 3명. 역할에 따라 리뷰 정책이 달라집니다.

FRONTEND

담당: 조현재

  • 리뷰어 없이 머지 진행 가능
  • 단, PR은 반드시 생성 (히스토리 · 변경 기록 목적)
  • PR 생성 후 본인이 셀프 머지
  • PR 본문에 작업 내용 요약은 필수
  • 이슈 자동 닫힘 위해 PR 본문에 Closes #N 권장
BACKEND · PIPELINE · 기타

FE 외 모든 영역

  • 3명 중 본인 제외 2명 모두에게 리뷰 요청
  • GitHub은 PR당 공식 Reviewer 등록은 1명만 가능
  • 나머지 1명은 PR 본문 또는 코멘트에 @멘션 으로 등록
  • 1명 이상이 Approve 하면 머지 가능 (전원 승인 강제 아님)
  • 코멘트만 달리고 Approve 안 된 상태는 머지 X
Review · Merge

리뷰 등록 + 머지 주체

리뷰 등록 방법

[PR 생성 화면]
  Reviewers: # 공식 등록 슬롯 — 1명만
    ☑ user-a

  Description: # PR 본문 또는 코멘트에 추가로 멘션
    ## #️⃣연관된 이슈
    Closes #42

    ## 📝작업 내용
    - X 기능 추가

    ## 💬 리뷰 요청
    @user-a (등록됨)
    @user-b (멘션으로 요청)

요약 표

작업자리뷰어 등록추가 멘션Approve 필요머지 주체
FE 조현재 없음 없음 없음 본인 (셀프)
BE 그 외 1명 1명 (@멘션) 1명 이상 본인

머지 주체 — 작성자 원칙

  • → PR 올린 당사자(작성자)가 직접 머지. 리뷰어가 대신 머지 X.
  • → 작성자가 최종 책임 (CI 통과 · 충돌 해결 · 머지 시점 결정).
  • → 머지 방식: Squash & Merge 권장 (히스토리 정리).
  • → 머지 후 브랜치는 자동 삭제 (GitHub 설정).
예외: 작성자가 휴가 등 사유로 머지 불가하면 코멘트로 위임 명시 후 리뷰어가 머지. 이 경우 PR 본문에 사유 기록.

Commit Message

Conventional Commits

제목은 짧고 명확하게. type: scope (optional) — 설명.

feat:     # 신규 기능
fix:      # 버그 수정
chore:    # 빌드 · 의존성 · 잡일
docs:     # 문서
refactor: # 리팩토링 (동작 변경 없음)
test:     # 테스트 추가/수정
perf:     # 성능 개선

# 예시
feat(gateway): add MCP tool auto-loading from registry
fix(catalog): resolve embedding search timeout
chore(deps): bump fastapi to 0.115

본문 (선택)

  • → 50자 제목 + 빈 줄 + 본문 (필요시)
  • → "why"가 비자명한 경우만 본문 작성
  • → 이슈 참조: Refs #42, Closes #42
