Protocol K
아키텍처 및 프레임워크 제안서

대상: Protocol K 공동 개발자 · 상태: 제안 (피드백 환영) · 작성일: 2026-05-18

한 줄 요약
TL;DR

메인 레포는 실시간 서비스(backend FastAPI + 변환된 OAS 저장소 + frontend 영역)를 묶고, 주기적 batch인 Conversion Pipeline은 별도 레포(protocol_k_pipeline)로 분리한다. Python 패키지는 uv, 인프라는 Terraform으로 구성하고 클라우드는 Azure 기준(팀원 의견에 따라 변경 가능)으로 작성했다. frontend 기술 스택은 담당자(조현재) 검토 영역이라 본 제안서에서는 다루지 않는다.

왜 모노레포 (메인)

실시간 서비스는 한 곳에서

Protocol K의 실시간 플랫폼 서비스(Gateway · Catalog UI)는 같은 OAS 스키마·tool 메타 모델·에러 포맷을 공유합니다. 별도 레포로 쪼개면 스키마 정의가 중복되거나 패키지 배포로 분리해야 합니다. 모노레포 = 한 PR에서 스키마 변경 + 클라이언트 반영을 동시에 검토 가능.

01

MCP Gateway (Backend)

변환된 tool을 MCP 프로토콜로 실시간 서빙

02

Catalog UI (Frontend)

변환된 tool을 검색·시도하는 사용자 인터페이스

03

변환된 OAS 저장소

pipeline 산출물(OAS YAML)을 git으로 관리

단, Conversion Pipeline은 별도 레포로 분리 — 실행 모델(batch vs realtime)·의존성 무게·배포 타깃이 메인 서비스와 다릅니다. 자세한 결정 근거는 아래 "Pipeline 분리" 섹션 참조.

메인 모노레포에서도 별도 레포가 합리적이 되는 시점

  • → 외부 사용자가 SDK로 의존 (Phase 4+)
  • → 컴포넌트별 보안 경계 분리 필요 (Phase 5)
  • → 그 전까지는 backend + frontend 모노레포 유지
Conversion Pipeline 분리

Pipeline은 별도 레포로

메인 레포는 실시간 플랫폼 서비스 담당. Conversion Pipeline은 주기적 batch로 동작 → 별도 레포(protocol_k_pipeline)로 분리합니다.

분리 근거 5가지

축메인 (protocol_k)파이프라인 (protocol_k_pipeline)
실행 모델 24/7 long-running Episodic batch (cron / Actions)
의존성 무게 FastAPI · pgvector (가벼움) LLM SDK + PDF/HWP 파서 (수백 MB)
배포 타깃 Azure Container Apps CLI / GitHub Actions / 별도 worker
릴리즈 주기 자주 (사용자 노출 기능) 가끔 (새 API 추가 시)
이미지 크기 service만 — 가벼움 변환 의존성 격리

분리 시 발생하는 비용

  • → 스키마 중복 위험 — OAS·MCP tool meta 형식을 양쪽이 알아야. 변경 시 양쪽 PR 필요.
  • → 타입 정의 공유 부담 — Pydantic 모델 일관성 유지 필요.

해결 방안 (3종 조합)

방안 A

OAS 저장소 = 메인 레포

pipeline 레포가 변환 후 메인 레포에 PR 자동 생성. service는 머지된 YAML 읽어서 tool 로드. = pipeline은 메인 레포의 contributor 역할.

방안 B

OAS 표준 포맷 활용

Protocol K 자체 스키마 최소화. Pydantic 모델은 service 쪽에 집중 유지. pipeline은 외부 표준(OAS 3.1)만 출력.

방안 C

Phase 2+ 스키마 패키지 분리

schema 변경이 잦아지면 작은 패키지(protocol_k_schemas)로 추출. 양쪽이 의존. Phase 1엔 도입 X.

두 레포 간 통신 흐름

[ protocol_k_pipeline ] [ protocol_k (main) ] ─ PDF / HWP 입력 ─ FastAPI Gateway ─ LLM 변환 → OAS YAML ─ Catalog UI ─ 카나리아 호출 검증 ─ tools/oas/*.yaml (commit 대상) ─ 결과 git PR 생성 ─────────────────────→ ─ PR 검수 → 머지 ─ service 재로드 → tool 노출

레포 구조 (최종)

  • → protocol_k (메인) — backend (FastAPI gateway + catalog) + frontend + tools/oas/ (변환된 OAS YAML 저장소)
  • → protocol_k_pipeline (별도) — PDF/HWP → OAS CLI/batch + 메인 레포 PR 자동 생성기

핵심 결정

  • → backend는 단일 uv 프로젝트. pipeline 의존성과 분리.
  • → frontend의 API 클라이언트는 backend의 OpenAPI에서 자동 생성 (openapi-typescript).
  • → tools/oas/가 service↔pipeline 의 경계. pipeline은 여기에 PR로 기여, service는 머지된 결과를 로드.
  • → 공유 스키마는 backend가 단일 소스 = OpenAPI Spec.
Backend

FastAPI + uv

비동기 IO · OpenAPI 자동 생성 · LangGraph/Pydantic 호환 · FastMCP 친화 · 학습 곡선 낮음.

FRAMEWORK

FastAPI 선택 이유

  • 비동기 IO— ASGI. 공공 API 동시 호출 적합
  • OpenAPI 자동 생성— 우리는 OAS 중심 시스템
  • Pydantic v2— 사용자 주 스택과 일치
  • FastMCP 통합— FastAPI 위에 자연스럽게 얹힘
  • 학습 곡선 낮음— 동료에게도 일반적

대안: Litestar (생태계 좁음), Node.js (LLM 생태계 거리감).

Frontend · 담당자 영역

본 제안서에서 다루지 않음

프론트엔드 기술 스택·구조·디자인 시스템은 담당자(조현재) 검토 영역으로 본 제안서에서는 다루지 않습니다. backend 가 제공하는 OpenAPI 명세를 클라이언트가 그대로 소비할 수 있도록 OAS 3.1 표준 형식으로 노출한다는 점만 명시합니다.

backend 접점 가이드
  • → API 스펙 — backend의 /openapi.json 자동 생성. 클라이언트는 이것을 소비.
  • → 인증 — Bearer token (사용자 키). CORS 정책은 backend가 설정.
  • → WebSocket / SSE — 필요 시 MCP 스트리밍 채널 별도 제공 (Phase 2+).
  • → 배포 타깃 — Azure Static Web Apps 등 (담당자 결정).
Data Layer · Azure 기준

Azure PostgreSQL + Redis + AI Search

역할별로 3종 저장소를 분리. Azure 매니지드 서비스 위주로 구성하여 운영 부담을 줄이고 백업·HA·모니터링을 Azure 표준에 위임합니다. 팀원 의견 조율 후 변경 가능.

용도저장소비고
Tool 메타데이터 · 사용자 키 · 호출 로그 Azure Database for PostgreSQL (Flexible Server) 관계형 데이터의 정본 저장소
캐시 · 세션 · 임시 데이터 · quota counter Azure Cache for Redis 정적 응답 캐시, 임베딩 결과 캐시, rate-limit
도구 description 임베딩 · 의미 검색 Azure AI Search (vector index) Azure 의 vector DB. ANN 검색 + 키워드 hybrid 검색 한 번에
변환된 OAS 원본 git (tools/oas/*.yaml) 캐싱 + diff 추적 + PR 검수 흐름
의견 조율 포인트
  • → AI Search 대신 pgvector 만 사용하는 안도 가능 (Phase 1 100개 규모는 충분).
  • → Redis 미도입 → PostgreSQL 자체 캐시 + 애플리케이션 메모리 캐시로 시작도 가능.
  • → AI Search 비용·러닝 커브 vs Postgres 단일화의 단순함 — 팀 합의 필요.
Infrastructure as Code

Terraform 으로 일원화

모든 클라우드 리소스(Container Apps · PostgreSQL · Redis · AI Search · Key Vault · Log Analytics)는 Terraform 으로 정의·관리합니다. 포털 클릭으로 만들지 않고, IaC 가 single source of truth.

구조 (예시)

infra/terraform/
├── envs/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── backend.tf      # tfstate = Azure Blob
│   └── prod/
│       └── ...
├── modules/
│   ├── container_apps/     # backend 호스팅
│   ├── postgres/           # Azure DB for PostgreSQL Flexible
│   ├── redis/              # Azure Cache for Redis
│   ├── ai_search/          # Azure AI Search (vector)
│   ├── key_vault/          # secret · DB 패스워드
│   ├── log_analytics/      # 로그 · 메트릭
│   └── network/            # VNet · private endpoint
└── README.md

핵심 원칙

  • → State backend = Azure Blob Storage. local state 금지. 잠금(state lock)으로 동시 apply 충돌 방지.
  • → 환경 분리 = envs/dev, envs/prod. variable 로 리소스 사이즈·tier 차등.
  • → Secret 관리 = Azure Key Vault. Terraform 은 KV 참조만, 실제 secret 은 KV 에서 회전.
  • → GitHub Actions OIDC 로 Azure 인증. Service Principal client_secret 을 GitHub Secret 에 박지 않음.
  • → CI 검증: PR 단계에서 terraform plan 자동, main 머지 시 apply. Drift detection 주기 실행.
Deployment · Azure 기준

GitHub Actions → Azure

클라우드는 Azure 기준으로 작성했습니다. 담당자가 Azure 환경에 익숙해 운영 비용·학습 곡선이 가장 낮다는 판단이지만, 팀원 의견에 따라 AWS · GCP · Cloudflare 등으로 변경 가능합니다.

배포 흐름

PR 머지 (main / dev) │ ▼ GitHub Actions 실행 (OIDC → Azure 인증) │ ├── infra ─→ terraform plan / apply │ ├── backend ─→ Docker build ─→ GHCR push ─→ Azure Container Apps revision 업데이트 │ └── frontend ─→ [담당자 결정] (별도 배포 파이프라인) ※ Pipeline (변환 batch) 은 별도 레포 protocol_k_pipeline 에서 자체 배포
