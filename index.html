National AI Competition · 2026

흩어진 공공 API
하나의 프로토콜로.

Protocol K는 한국 공공 Open API의 파편화된 명세·인증·포맷을 MCP(Model Context Protocol)로 일원화하고, 변환된 도구를 검색·발견 가능하게 만드는 인프라 플랫폼입니다. 이번 1차 범위는 변환 + 검색에 한정합니다. 에이전트·워크플로 자동화는 그 다음입니다.

현황 — Why this matters

10만+ 공공 API, 활용률은 미미

공공데이터포털(data.go.kr)에 등록된 데이터는 10만 건을 넘지만, 기계 가독 명세(OpenAPI Spec)를 제공하는 API는 20% 미만으로 추정됩니다. 대부분의 공공 API는 LLM 에이전트가 즉시 사용할 수 없는 형태로 방치되어 있습니다.

01

인증 키 이름이 4종

serviceKey, confmKey, apiKey, KEY — 단일 포털 안에서 표준이 없습니다.

02

응답 포맷 XML 50~60%

dataType, type, resultType, format — JSON 전환 파라미터 이름조차 통일되어 있지 않습니다.

03

명세는 PDF·HWP·DOCX

활용가이드 대부분이 PDF·HWP·DOCX. 자동 코드 생성도, LLM tool 변환도 원천 차단되어 수기 클라이언트 작성 외에는 길이 없습니다.

04

기존 MCP 시도 = 0.01%

최대 규모 오픈소스(Koomook/data-go-mcp, 255★)도 6개 API만 커버. 전부 1 API = 1 서버의 수작업 래핑입니다.

05

네이밍·페이지네이션 카오스

base_date(snake) · pageNo(camel) · LAWD_CD(SCREAMING)가 한 포털 안에 공존. 페이지네이션도 제각각입니다.

06

에러 표현 비일관

HTTP 200 OK + body 안 에러 코드, resultCode 값이 API마다 다름(00/OK/NORMAL_SERVICE).

솔루션 — What we build

변환 + 검색. 그 다음이 에이전트.

Protocol K의 1차 가치는 두 가지로 정확히 한정됩니다. 에이전트, Composite, Tool Router, Workflow Composer 같은 상위 기능은 이 인프라 위에 자연스럽게 올라가는 2차 단계입니다. 견고한 1단계 없이는 그 위 어떤 기능도 모래성입니다.

Phase 1 — 이번 대회 범위

MCP-ify + Catalog Search

  • 다양한 포맷의 공공 API를 단일 MCP tool로 변환하는 파이프라인
  • 변환된 도구의 키워드 + 의미 검색 (임베딩 색인)
  • 인증 키·응답 포맷·에러 정규화 게이트웨이
  • 도구 상세 페이지 + 인터랙티브 Try-it 호출
  • 변환 비용·자동화율·카나리아 통과율 등 KPI 추적
Phase 2+ — Roadmap

그 위에 올라가는 것들

핵심 분석 — Conversion feasibility

공공 API 4-Tier 분류

Tier별 해결 지점 — Problems to solve

각 Tier가 궁극적으로 100% 동작하려면

T1·T2·T3 모두 궁극(Phase 5)에는 자동 변환 가능해야 합니다. 각 Tier가 "쉽다 / 어렵다"는 평균치일 뿐, 실제로는 각각 풀어야 할 구체 문제들이 있습니다. 아래는 Phase 1~5에 걸쳐 점진적으로 해결될 문제 목록입니다.

TIER 1

표면적 쉬움 · 함정 5

REST + JSON + OAS가 있다고 끝이 아님.
  1. OAS 품질 편차 description 한글·누락·모호. LLM 메타데이터 보강기로 영문 description·예시 자동 생성.
  2. 인증 위치 변형 같은 odcloud 안에도 header 인증과 query 인증이 혼재. Auth Mapper에서 위치 추상화.
  3. MCP tool 이름 충돌 자동 생성 시 도구명 중복. 카테고리 prefix 규칙(weather.short_forecast) 강제.
  4. OpenAPI 2.0 vs 3.1 공공 OAS 일부 Swagger 2.0. OAS 3.1로 자동 컨버터 내장.
  5. 에러 body 비표준 JSON 응답이라도 에러 시 body 다름. T2와 공유하는 에러 정규화 미들웨어로 처리.
TIER 2 · 메인

어댑터 일반화 · 6대 도전

공공 API 70%가 여기. 해결되면 사실상 끝.
  1. PDF/HWP → OAS 추출 정확도 ⚠ 표 구조 보존 파서 + LLM + 사람 검수 UI. 핵심 위험. Phase 1 첫 2주 PoC.
  2. XML → JSON 응답 변환 명세 기반 매핑 함수. 한글 필드명(efcyQesitm) → 영어 표준 키 LLM 매핑.
  3. 함정 파라미터 (격자좌표·법정동·통계분류) Phase 1: 입력 description에 명시. Phase 2+: Resolver tool로 자동 변환.
  4. 에러 정규화 HTTP 200 + body 안 resultCode → 표준 HTTP status + {error:{code,msg}}.
  5. 페이지네이션 4종 통일 pageNo+numOfRows / currentPage+countPerPage / objL* → 표준 cursor+limit.
  6. 인증 키 다양성 + URL 인코딩 serviceKey/confmKey/apiKey + 인코딩/디코딩 키 두 종. 백엔드별 메타데이터 + 게이트웨이 자동 처리.
TIER 3

커스텀 핸들러 · 5 도전

Phase 4부터 본격. 라이브러리화가 관건.
  1. SOAP 추출 WSDL → OAS 변환기 또는 SOAP 직접 어댑터.
  2. HMAC / 세션 / OAuth 토큰 갱신 백그라운드 잡 + 보안 저장소.
  3. 파일 응답 (ZIP / CSV / HWP) 다운로드 → 압축 해제 → 파싱 → JSON 변환을 단일 tool 안에 캡슐화.
  4. Stateful 멀티-스텝 흐름 여러 API 호출을 단일 MCP tool 내부 워크플로로 hardcode.
  5. bulk dump 크기 제어 streaming / 페이징 / 캐싱 정책으로 응답 사이즈 관리.

핵심 인사이트. T2 6개 문제 풀면 공공 API 70% 흡수. T3 5개 풀면 90% 도달. T1은 T2 인프라(에러 정규화·tool 명명·인증 매핑) 위에서 거의 자동으로 흡수됨.

구현 계획 — Phase 1

1단계 = 프로젝트 난이도의 90%

에이전트·Composite·Router 같은 상위 기능은 의도적으로 미룹니다. Phase 1에서 변환·등록·검색을 끝까지 완성하는 것이 이 프로젝트의 핵심 가치입니다. 4개 모듈로 분할하여 병렬 진행 가능합니다.

Phase 1 큐레이션 — 코어 30개

실제로 변환할 30개 API

무작위 30개가 아닌, 4개 기준으로 선정한 큐레이션 리스트입니다. (1) 활용신청 누적 상위, (2) 다양한 Tier 분포, (3) 다양한 카테고리, (4) 다양한 함정 패턴. 이 30개 안에 함정 패턴이 거의 다 포함되어 있어, 변환에 성공하면 어댑터의 일반성이 증명됩니다.

왜 이 30개인가. 이 안에 함정 패턴이 거의 다 포함됩니다 — XML default, 격자좌표, 법정동코드, confmKey, apiKey, POST+JSON body, 한글 필드명, 다단 분류, bulk 응답 등. 이 30개 어댑터를 일반화하면 Phase 2의 200개로 확장하는 비용이 급격히 떨어집니다.

확장 로드맵 — 5 Phases

30 → 9,000개로 가는 길

각 Phase = 단순히 "더 많은 API"가 아니라 "새로운 문제를 풀어야 가능한 다음 단계". Phase 1~5 각각이 다른 기술·운영·제품·데이터·비즈니스 챌린지를 가집니다.
